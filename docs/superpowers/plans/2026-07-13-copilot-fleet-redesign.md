# Copilot Fleet Redesign — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Rebuild this repo's fleet as a VS Code Copilot agent plugin — 5 agents / 26 skills — from first principles, fixing the audit's six Tier-2 bugs during the port and never porting them as-is.

**Architecture:** Seven phases, with a blocking cross-runtime preflight before the first fleet agent: one canonical fleet manifest + canonical bodies generate distinct Copilot and Claude wrappers, then content proceeds canonical agent definitions → dependency slices that make those definitions runtime-ready → observability skills → machinery → distribution → pilot → rollout. An incomplete production agent is never projected into either runtime. Each phase is its own branch off `main`, opened and closed with the Section 0 run protocol; a phase is not done until audits A–C are green (D runs over the content-complete fleet and again at pilot). Probes and tests are written first and failing, scoped to the artifact under change.

**Tech Stack:** canonical JSON + Markdown bodies; generated Copilot `.agent.md` and Claude `.md` wrappers; shared `SKILL.md` bundles; Python 3 stdlib scripts (`scripts/generate_fleet.py`, `scripts/gate_a.py` and its steps); PowerShell + POSIX sh (`setup.ps1`/`setup.sh`); GitHub Actions; VS Code Copilot chat and Claude Code as separately validated runtimes.

**Design spec:** `docs/superpowers/specs/2026-07-13-copilot-fleet-redesign-design.md` (Status: approved). Evidence base for every content fix: `docs/AUDIT-2026-07-12.md`. Harvest source: the configurable, origin-verified `SDE_AGENTS_SOURCE` checkout pinned to a full fetched `latent-sre/sde-agents` `origin/main` SHA (one-way object copies; this repo then owns them). A local path is an operator input, never provenance.

## Global Constraints

- **Run protocol (spec Section 0) opens and closes every phase.** Open:

  ```
  git status --porcelain                   # must be empty
  git fetch --prune origin
  git switch main && git pull --ff-only origin main
  git log --oneline -1                     # record the SHA this phase branched from
  git switch -c <phase-branch>             # branch from main, NEVER from another phase branch
  ```

  Before opening each PR (the owner merges PRs mid-session — `main` moves under you):

  ```
  git fetch --prune origin
  git rebase origin/main
  git log --oneline origin/main..HEAD      # assert: ONLY this phase's commits
  ```

  Close: audits **A** (`py -3 scripts/gate_a.py`), **B** (collapsed into A via `scripts/test_no_regressions.py` from Task 2 onward), **C** (three independent reviewers — code, security, and one briefed **only on the spec** checking conformance). **D** runs at the first content-complete point (Task 36) and at pilot (Task 48) — never over a half-built fleet.
- **Python is `py -3` on this machine, NOT `python3`** (`python3` is the Microsoft Store stub). Every gate goes through `scripts/gate_a.py`, which re-invokes sub-steps under `sys.executable` — do not hardcode any interpreter name into a new script's docs or CI beyond what Task 1 shows.
- **Shell for plan commands is explicit.** Unlabeled fenced shell/SPC commands are POSIX and run under Git for Windows Bash on this machine; phase open must prove `bash --version`, `git`, `grep`, `sort`, and `comm` before executing them. PowerShell commands are fenced/labeled `powershell`; Python gates remain `py -3`. A missing required shell/tool is a STOP—never translate a load-bearing count/diff ad hoc in the middle of a phase.
- **Verbatim-move discipline.** Ported prose relocates unchanged — the plan names the source file and section, it never re-types the prose (re-typing invites silent transcription drift). **The single stated exception (spec Section 5d): every pointer to a bundled file (`references/`, `assets/`, `scripts/` inside a skill) is rewritten to a relative Markdown link during the move** — `[forms](./references/forms.md)`, never `` `references/forms.md` ``. Predicate tables keep their shape; the right-hand cell becomes a link. Every bundled file must be linked from the body (unlinked = never loads = delete it). Each port task lists its exact pointer-rewrite worklist; `scripts/check_links.py` (Task 9) enforces it mechanically from the first port onward.
- **The six Tier-2 bugs are fixed during the port, never ported as-is** (audit §2.1–2.6): blue-green name rotation · WQL `by`-clause deletion · SPL `timechart` bucketing · bare `cf auth` (no argv) · `error_budget.py` severity + window-pair binding · Grafana Enterprise-licence facts. `scripts/test_no_regressions.py` (Task 2) makes porting any of them a Gate-A failure.
- **One source of truth, runtime-specific generated projections.** Authors edit agent metadata in `canonical/fleet.json`, agent bodies in `canonical/agents/*.md`, and command bodies in `canonical/commands/*.md`. `scripts/generate_fleet.py` emits root `plugin.json` + `generated/copilot/agents/*.agent.md` + plugin commands under `generated/copilot/commands/*.md` + fallback prompt wrappers under `generated/copilot/prompts/*.prompt.md`; it separately emits `.claude-plugin/plugin.json` + `generated/claude/{agents,commands}/*.md`. It also emits `.plugin/plugin.json` as an exact-byte compatibility alias of root `plugin.json`: this binds current VS Code native plugin selection to the Copilot projection when the Claude marker is present, without creating a third semantic projection or authoring source. `--check` owns the complete `.plugin/` directory and rejects missing, stale, linked, hardlinked, or unexpected selector content. Both semantic manifests share generated name/version. Root Copilot directory fields and Claude explicit-file lists stay runtime-specific.
- **Runtime boundaries are structural.** Shared skills/references stay under `skills/`, but the exact 26-name skill catalog and each active skill's bundled-file inventory (`references/`, `assets/`, and `scripts/`) live in `canonical/fleet.json`; unexpected runtime-visible skill directories or bundled files fail `--check`. Agent entries carry one fixed `required_skills` list. Root `hooks.json` is Copilot's auto-discovered hook; `hooks/hooks.json` is Claude's. Neither manifest references a hook file. Every agent is explicitly named. A nonempty Copilot `agents:` list automatically adds the `agent` tool; Claude delegation maps to `Agent(target)` and records the nested target-list degradation; handoffs emit only for Copilot; model fields map per runtime.
- **Construction state is not runtime permission.** Every catalog skill is exactly `planned` (name + activation task, no directory) or `active` (name + exact directory/bundle inventory). A planned skill directory on disk, or an active record without exact on-disk parity, fails generation. An agent definition may name planned catalog entries, but it is **runtime-ready only when every `required_skills` entry is active and the ready set is closed over every delegation and handoff target**. The generator computes the greatest closed ready set, emits wrappers and manifest entries only for that set, and recursively rejects stale/unexpected wrappers for every unready agent. `required_skills` is integrity metadata, not a runtime preload/allowlist: it is never copied to Copilot, never emitted as Claude `skills:`, and only causes the generic Claude `Skill` tool to be granted once the agent is runtime-ready. Each marked agent-body dependency names both registered identities—Copilot `<skill>` and Claude `sre-agents:<skill>`—and generation rejects a missing, wrong, or duplicate identity pair. Canonical `skill_dependencies` separately records mandatory skill-to-skill loads; an active skill may depend only on active catalog entries and its marked dependency block must carry the same paired identities. `assembly_state` is `boundary-only` in Phase 1, `content-building` in Phases 2–3, and `content-complete` only with 26 active skills, all five agents ready, every skill dependency resolved, and no planned catalog entry. Task 33 permanently adds `--require-content-complete` to Gate A so later phases cannot downgrade this contract.
- **Never register a mutable construction tree.** Tasks 1–39 author with native-plugin, fallback, and Claude production discovery disabled. Per-file atomic replacement is a filesystem-safety property, not a multi-file publication transaction. A runtime checkpoint first finishes `--check`, commits or otherwise durably binds one exact tree, copies it into an immutable disposable snapshot outside every authoring path, records source-commit/tree and runtime-tree digests, then enables exactly one channel in a neutral consumer workspace/profile; Diagnostics must identify only that snapshot and account for any intentional collision fixture. Disable/unregister it before testing the next channel and remove it afterward. Any runtime-visible or tested-input byte change requires a new snapshot and invalidates affected evidence. Append-only sanitized/hash-bound evidence outside the runtime inventory may be added afterward only when it names the tested source/tree provenance, carries a durable checked-in path→hash binding when merge methods may discard the intermediate object, and the final runtime digest remains identical. Before Task 40 exists, these snapshots are construction evidence only, never promotion evidence; release-bound evidence later uses the protected canary state machine.
- **The `tools:` vocabulary is UNVERIFIED and is the primary safety control.** Task 2b's five-assertion boundary check plus four separate negative-capability prompts run in real VS Code Copilot and Claude before any production definition is authored. Task 26 repeats the terminal-agent assertions against the first runtime-ready production cohort before any normal production-agent prompt. The STOP conditions are non-negotiable; on any failure, amend spec Section 5 before proceeding.
- **Names are kebab-case** (silent-load-failure class). **Descriptions carry verbatim user phrasings** (`Triggers: "..."`) plus boundary clauses. The deterministic CI cap is **≤600 UTF-8 bytes per description** (the ≈150-token design target); model-token counts are reported separately and never use a chars fallback as a gate. The one measured-good model is old `agent-authoring`'s description (3/3 baseline, twice).
- **Evidence labeling is uniform doctrine**: `[verified]` / `[sourced]` / `[unverified]`, using the sde-agents canonical stems (they become validator-v2 checks in Task 37). Gate C rule: **every stack-specific command, query, or API field in a ported or new skill is either executed against the real system and labeled `[verified]`, or labeled `[unverified]`.** An unlabeled claim is a review finding.
- **Probes/tests first and failing, scoped to the artifact under change.** This governs each phase's *internal* order — it does not mean writing the whole suite before the whole fleet.
- **`legacy/claude-fleet/` is frozen after Task 1.** Ports copy *from* it; nothing edits it. (It is also what `test_no_regressions.py` self-arms against — retiring it is a Phase 7 step with a named consequence.)
- **A skill never transcribes an artifact that lives in the repo — point at it** (anti-rot doctrine, audit through-line).
- **Do not transcribe `gate_a.py`'s steps into any document** — the doc drifts; the command is the truth.
- **Sister-repo provenance:** set configurable `SDE_AGENTS_SOURCE` (the local path is not part of the contract), require its `origin` to normalize to `latent-sre/sde-agents`, run `git -C "$SDE_AGENTS_SOURCE" fetch --prune origin`, and pin the full `git -C "$SDE_AGENTS_SOURCE" rev-parse origin/main` SHA once at each phase open. Record remote URL + SHA in the phase PR. Every `SDE:` source and absolute source citation below is logical: read bytes only with `git -C "$SDE_AGENTS_SOURCE" show "$SDE_AGENTS_SHA":<path>` or a detached/read-only archive of that exact SHA. Never copy/read the mutable sister worktree, and never fall back to stale local `HEAD` when fetch or identity verification fails. SPC-1 uses the same pinned object; a mid-phase `origin/main` move does not change the harvest.
- **One release state machine—no feature-to-release edge.** Once Task 40 installs the trusted harness, every unpublished plugin/runtime/setup state that needs release-bound behavioral evidence is exercised only from a unique, immutable, temporary `canary/<phase>/<full-sha>` ref. Tasks 26, 33, and all runtime/eval checkpoints in Tasks 35–39 predate that harness and use local immutable construction snapshots only; their evidence can close construction gates but can never authorize promotion. Canary CI is dispatched from a workflow definition pinned to protected `main`, never from the candidate ref; the unreviewed checkout receives no secrets or writable token. A canary ref is never a setup/refresh source and never changes in place: a code change gets a new ref and new evidence. Repository changes then go through a Code-Owner-reviewed PR into `main`. Because merge/squash/rebase may change the commit SHA, every promotion freezes current `main` and repeats a short final canary/cleanup at the **exact merged-main SHA**; that run, not merely the pre-merge run, is the workflow's promotion evidence. A protected promotion workflow, approved through the `fleet-release` environment and authenticated only as the dedicated promotion GitHub App, creates or fast-forwards `release` to that identical reviewed green `origin/main` SHA; it creates no merge/squash/rebase commit. Layered no-bypass rules still deny force-push and deletion to that App. Human/admin direct writes, force-pushes, resets, and direct reverts on `release` are forbidden. Pilot fixes and roll-forward rollback use the same pre-review canary → PR-to-`main` → exact-main canary → exact-SHA-promotion path.

## On "paste the section verbatim" steps and new stack content

This plan does not reproduce ported prose inside itself: for a move, "relocate `## X` from `<file>`, unchanged" is more exact and safer than re-typing it here. `scripts/generate_fleet.py --write` emits only fields documented for each runtime; it must not blindly copy canonical metadata into both schemas. The canonical `skills` list is a construction catalog, while only its `state: active` entries are runtime-visible. With no runtime-ready agents or commands, both manifests emit explicit `"agents": []` and `"commands": []`; those fields replace their runtimes' default discovery paths. With no active skills, both emit `"skills": []` as an explicit schema state, but **Claude's `skills` field is additive and does not suppress its default `skills/` scan**: the actual control is the generator's exact empty active skill-tree inventory. Once the first closed cohort becomes runtime-ready, root `plugin.json` uses `"agents": "./generated/copilot/agents/"`, while `.claude-plugin/plugin.json` explicitly lists only ready generated-Claude agent files; once the first skill becomes active, both use `"skills": "./skills/"`. **Task 1 implements only the zero-command scaffold, default suppression, and orphan rejection; any nonempty production command is a hard `format boundary not accepted` error.** Only after Task 2b's live rows and accepted-evidence gate does Step 5 port the nonempty renderer/constants/tests described next. A nonempty canonical command record then has exact keys `name`, `source`, nonblank `description`, `argument_mode` (`required` or `none`), `argument_usage` (nonblank only when required), and `invocation_mode: manual`. A required-argument canonical body contains the runtime-neutral sentinel `{{arguments}}` exactly once; a no-argument body contains none. The Task-2b spike pins each runtime's supported description metadata, argument expression, and manual-only control. The accepted production generator emits three drift-joined views—Copilot-plugin `generated/copilot/commands/<name>.md`, VS Code fallback `generated/copilot/prompts/<name>.prompt.md`, and Claude `generated/claude/commands/<name>.md`—mapping only documented metadata, translating the sentinel, and emitting `disable-model-invocation: true` where supported. VS Code fallback prompt files are inherently user-invoked; compatibility evidence must pin that behavior rather than emit an unknown field. Drift parity means: strip the exact generated frontmatter for that view, normalize its one argument expression back to `{{arguments}}`, then require byte equality with the canonical body. A runtime that cannot carry required arguments **or cannot prove manual-only invocation** is a STOP, not a degradation. If a runtime schema has no description field, `compatibility.json` records intentional non-emission and tests it; blank, silently ignored, duplicated, or drifted metadata fails. Root Copilot points `commands` at the generated Copilot command directory; Claude enumerates the generated Claude files. Both carry identical generated `name`/`version` and contain **no hook path**. Production `assembly_state: boundary-only` always requires canonical `commands: []`, no canonical command bodies, empty/absent generated command/prompt directories, and manifest `commands: []`; only the isolated spike may be nonempty in Phase 1.

The production generator permits the zero-ready-agent/zero-active-skill/zero-command scaffold, rejects default-path bypass files, and enforces exact canonical-body parity across all definitions plus exact wrapper parity for the derived ready set.
