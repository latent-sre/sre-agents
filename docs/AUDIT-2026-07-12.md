# Fleet audit — 2026-07-12

A full review of the fleet (9 agents, 37 skills, plus machinery) by five parallel reviewers, with
every load-bearing claim re-verified by hand. This document is the **evidence base** for the fixes
that followed and for the ones still outstanding.

**Why it exists:** the Copilot redesign spec
(`docs/superpowers/specs/2026-07-13-copilot-fleet-redesign-design.md`) repeatedly says "with audit
fixes applied" and "factually clean per audit." Without this document those claims are
unfalsifiable, and an implementer porting a skill would either re-port the bug or invent a fix. A
finding that lives only in a chat transcript is not a finding.

Status key: **OPEN** (bug is live on `main` today) · **FIXED** (PR merged, cited).

---

## The through-line

Almost every serious defect had one shape: **a check that reports success without executing the
thing it names.** That is the same failure mode as the Windows-guard bug the README is built around
("the guard never ran, no decision was emitted, and Claude Code let the command through"),
reproduced one layer up — in the machinery meant to catch it.

A second pattern drives the content bugs: **prose that transcribes an artifact rots the moment the
artifact changes.** `agent-security` transcribed the guard's behavior and drifted from it; `AGENTS.md`
transcribed the tool census and omitted `WebFetch` entirely. The durable rule, now doctrine: **a skill
must never transcribe an artifact that lives in the repo — point at it.**

## Tier 2 — content bugs that would burn an on-call (ALL OPEN)

These are live in the fleet on `main` today. Each is either fixed in place, or fixed during the
port — but never ported as-is.

### 2.1 `pcf-deploy` — the blue-green playbook is a one-shot that silently inverts — **OPEN**

`.claude/skills/pcf-deploy/SKILL.md`, § "Blue-green (classic) — preferred for prod".

The app names `checkout-blue` / `checkout-green` are **never rotated**:

- **Cycle 1:** `checkout-blue` does not exist (nothing in the playbook or manifest ever creates it), so
  `cf unmap-route checkout-blue` fails, and the documented rollback ("re-map the prod route to
  `checkout-blue`") has no target.
- **Cycle 2:** `checkout-green` *is* the live prod app — step 3 mapped the prod route onto it.
  `cf push checkout-green -f manifest.yml --no-route` now pushes **onto the app serving production**.
  `--no-route` does **not** remove a route the app already holds: *"All declared routes are ignored.
  This flag does not unbind all existing routes associated with the app."*
  (docs.cloudfoundry.org/devguide/deploy-apps/manifest-attributes.html) So this is an in-place,
  all-instances restart of the app taking production traffic, with no green to test and no blue to
  roll back to.

**Fix:** keep a stable live name (`checkout`) and make green the disposable one; rotate names after the
soak (`cf delete checkout -f && cf rename checkout-green checkout`).

**Severity:** the flow is marked *"preferred for prod"*, and its failure mode is an unannounced
in-place production deploy with no rollback — during the exact change the playbook exists to de-risk.

### 2.2 `wavefront-queries` — a fabricated WQL `by` clause — **OPEN**

`.claude/skills/wavefront-queries/SKILL.md` teaches `sum(ts(app.http.requests.count)) by (app)` as an
"equivalent `by` keyword" form, and adds that *"the `by` form is valid but requires parentheses around
the grouping keys."*

**There is no `by` form in WQL.** Documented syntax groups by a trailing parameter:
`sum(<tsExpression>[, metrics|sources|sourceTags|pointTags|<pointTagKey>])`
(docs.wavefront.com/ts_sum.html). The only `by` construct in the language reference is series matching
inside `join()`.

**Fix:** delete the `by` form; state plainly that WQL has no PromQL-style `by` clause.

**Severity:** the query errors out — but worse, the invented "requires parentheses" caveat reads like
hard-won knowledge and makes an engineer *more* confident in a query that cannot run. Confabulated
detail is the highest-risk class in an ops skill.

### 2.3 `splunk-triage` — the 3-sigma query filters before bucketing — **OPEN**

`.claude/skills/splunk-triage/SKILL.md`, the anomaly-detection example:

```spl
| where status>=500
| bin _time span=5m
| stats count by _time
| streamstats window=12 current=f avg(count) AS baseline stdev(count) AS sd   ```trailing 1h```
```

`stats count by _time` emits a row only for groups **present in the data**. Because the `where` runs
first, a 5-minute bucket with zero errors produces **no row at all**. Therefore:

- `window=12` is 12 *error-containing rows*, not 12 × 5 min — on a healthy service that errors in one
  bucket in twenty, those 12 rows span ~20 hours. The inline comment "trailing 1h" is false.
- The baseline is the mean/stdev of buckets **that already had errors** — systematically far above the
  true (mostly-zero) baseline — so `count > baseline + 3*sd` **under-fires** on the spike it exists to
  detect.

**Fix:** bucket first, filter inside the aggregation — `timechart` fills empty buckets, `stats` does not:

```spl
| timechart span=5m count(eval(status>=500)) AS errors
| streamstats window=12 current=f avg(errors) AS baseline stdev(errors) AS sd
| where isnotnull(baseline) AND sd>0 AND errors > baseline + 3*sd
```

**Severity:** silently under-detects. Notable because it is the same contamination class the skill's own
"Three traps here" section claims to have fixed.

### 2.4 `slo-error-budget` — `error_budget.py` reports "within budget" at 5.5× burn — **OPEN**

`.claude/skills/slo-error-budget/scripts/error_budget.py`. The severity ladder takes
`both = min(burn_long, burn_short)`; the final `else` is reached whenever `min(...) < 1` **and both
windows are under 6×** — so a long window burning at 5.5× lands in a branch labeled "within budget
(burn < 1x)". Reproduced:

```
$ python error_budget.py --slo 99.9 --sli-long 99.45 --sli-short 99.95
  burn (1h):  SLI 99.45%  ->  5.50x
  burn (5m):  SLI 99.95%  ->  0.50x
  severity: within budget (burn < 1x)          <-- false all-clear
  at the 1h rate, the full 28d budget is gone in 5.09 d (7331 min)
```

The tool contradicts itself two lines apart: *"gone in 5 days"* and *"within budget."*

**Second defect, same file:** `--long-window` / `--short-window` are **cosmetic strings** — the
14.4 / 6 / 1 burn-rate thresholds are applied to whatever window pair you hand it. A 2× burn measured
over one hour consumes 0.28% of a 30-day budget (noise) and the tool cuts a ticket. The skill's own
table correctly binds 1× to a **3d + 6h** pair; the script ignores the binding. (Google SRE Workbook,
*Alerting on SLOs*: the burn rate and its window pair are a single unit.)

**Fix:** the `else` must say what is true — no *multi-window* alert, which is not "healthy"; and the
window pair must be a validated input that selects the threshold.

**Severity:** this is the one output an on-call reads under pressure, and it is a false negative in the
dangerous direction.

### 2.5 `github-actions-ci` — `cf auth` leaks the prod password into argv — **OPEN**

`.claude/skills/github-actions-ci/SKILL.md`, under a heading that says **"(paste-ready)"**:

```yaml
env:
  CF_USERNAME: ${{ secrets.CF_USERNAME }}
  CF_PASSWORD: ${{ secrets.CF_PASSWORD }}   # "residual argv exposure during cf auth"
run: |
  cf auth "$CF_USERNAME" "$CF_PASSWORD"
```

The step **already sets both as environment variables**, which is exactly how `cf auth` is designed to
be fed. Passing them again as positional arguments is what *creates* the argv exposure. The cf CLI's
own help says so: *"Providing your password as a command line option is highly discouraged… Consider
using the CF_PASSWORD environment variable instead"* (`command/v7/auth_command.go`).

**Fix:** `cf auth` — with no arguments. The skill currently teaches an operator to *accept* a
production-credential risk that a one-word edit removes.

### 2.6 `grafana-dashboards` — the data sources are Enterprise-licensed, and one doesn't exist — **OPEN**

Verified against the Grafana plugin catalog API:
- `grafana-wavefront-datasource` → `"status": "enterprise"`
- `grafana-splunk-datasource` → `"status": "enterprise"`
- ThousandEyes → **404, no Grafana data-source plugin exists**

The skill builds its entire data-source story on all three without saying any of this. An engineer
follows `grafana-dashboards` → `wavefront-queries`, writes `ts()`, and discovers at build time either
that the plugin needs a licence they don't have, or — on the vendor's own OSS path — that the panel
only accepts **PromQL**, not the WQL the sibling skill just taught.

## Tier 3 — discoverability (measured; superseded in part)

Recorded probe baselines in `evals/discovery/` showed **`agent-security`, `ops-cli`, `pcf-deploy`, and
`sde-ladder` all recorded `saw: none`** — no skill loaded at all, on prompts written specifically to
trigger them. That `agent-security` is among them is the sharp part: `AGENTS.md` documents a live
lethal-trifecta exposure in this very fleet, and the skill that reasons about it never loads.

**Root cause:** 36 of 37 descriptions carried **zero verbatim user phrasings**. The one that does —
`agent-authoring` (`Triggers: "write me an agent", "my skill never triggers"…`) — baselines 3/3, twice.

**Partly superseded (2026-07-13):** the clean-room rig (PR #52) proved the old baselines were measured
on a machine whose personal global skills competed with the fleet, and that the contamination cut
**both ways** — it suppressed `sde-ladder` *and flattered* `sre-ladder`. Clean, both ladders sit at the
same rate. Any number in the pre-clean-room notes must be re-measured before it is acted on. The
`saw: none` results are *not* explained by shadowing (nothing fired at all) and stand.

Also found: `pcf-deploy` and `bamboo-to-actions-migration` set `disable-model-invocation: true` yet had
discovery scenarios expecting autonomous invocation — **unpassable by construction**, and neither the
validator nor the probe knew what that flag was.

## Verification method

Findings were produced by five parallel reviewers (roster/handoffs, skill triggers, process-skill
bodies, stack-skill correctness, eval machinery) and then **independently re-verified by hand** before
any was acted on. Load-bearing claims were reproduced, not quoted: the guard bypasses were piped
through the shipped guard; the prod-gate false positive was run against the real graders; the
error-budget all-clear was reproduced from the CLI; the Grafana plugin licences were fetched from the
catalog API; the `--no-route` and `cf auth` semantics were read from CloudFoundry docs and cf CLI
source.

Claims that could **not** be verified are labeled as such and were not acted on — notably
`pcf-deploy`'s "revisions are GA in cf CLI v8.10.0+" (no source found either way), and the
`matcher: WebFetch` hook mechanism (documented, never executed).
